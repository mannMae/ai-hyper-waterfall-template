## 3. 작업지시자-에이전트 협업 모델

### 3.1 역할

| 역할 | 담당 | 책임 |
|------|------|------|
| **작업지시자** (Human) | 프로젝트 오너 | 타스크 지정, 비즈니스 가치 검증, 승인/반려, 방향 수정 |
| **Antigravity** (AI 에이전트) | AI 개발자 | 코드 분석, 구현, 테스트, 문서 작성, 디버깅 |

### 3.2 타스크 진행 절차 (Hyper-Waterfall)

1. **작업지시자**: 타스크 지정 (`mydocs/orders/`에 등록)
2. **Antigravity**: 수행 계획서 작성 → 승인 요청
3. **작업지시자**: 승인 또는 수정 지시
4. **Antigravity**: 구현 계획서 작성 (상세 단계 포함) → 승인 요청
5. **작업지시자**: 승인
6. **Antigravity**: 단계별 구현 → 각 단계 완료 보고서 → 승인 요청
7. **작업지시자**: 검증 후 승인 또는 피드백
8. **반복...**
9. **Antigravity**: 최종 결과 보고서 작성 → 승인 후 브랜치 병합

### 3.3 핵심 원칙

- **작업지시자의 지휘권**: Antigravity가 제안할 수 있으나, 최종 결정과 책임은 작업지시자에게 있습니다.
- **즉각적 피드백**: "이 방식은 틀렸습니다", "이전으로 되돌리세요" 등 즉각적인 교정이 생산성을 높입니다.
- **승인 기반 진행**: 단계별 승인 없이 임의로 다음 단계나 타스크로 넘어가지 않습니다.
- **정량적 검증**: 주관적인 판단보다는 데이터, 로그, 테스트 결과 등 객관적 지표로 성공 여부를 판단합니다.

## 4. 디버깅 및 분석 프로토콜

작업지시자와 Antigravity가 **동일한 언어로 문제를 식별**할 수 있는 도구 체계가 중요합니다.

### 4.1 정량적 식별자 활용

문제를 지칭할 때 "그 부분"이라는 표현 대신 고유 ID, 행 번호, 메모리 주소, 또는 시각적 좌표 등을 사용합니다.

- **나쁜 예**: "하단 버튼의 색상이 이상합니다"
- **좋은 예**: "`Button#submit-01`의 배경색이 `#FF0000`이어야 하는데 `#CC0000`으로 렌더링됩니다"

### 4.2 덤프 및 로그 도구

Antigravity는 시스템의 내부 상태를 즉시 확인할 수 있는 덤프(Dump) 도구나 상세 로그를 적극적으로 활용해야 합니다.

## 5. 피드백 체계

| 유형 | 예시 | Antigravity 대응 |
|------|------|-------------|
| **즉시 수정/복구** | "원복하세요" | 변경 사항 즉시 롤백 (`git checkout` 등) |
| **방향 전환** | "이 접근은 비효율적입니다" | 현재 접근 폐기, 새로운 분석 및 계획 수립 |
| **정보 제공** | "정확한 임계값은 0.5입니다" | 제공된 수치를 바탕으로 로직 정밀 보정 |
| **백로그 등록** | "나중에 개선합시다" | 현재 범위를 확정하고, 개선안을 별도 이슈로 등록 |

## 7. Git 워크플로우

- **main**: 안정 버전 (배포용)
- **develop**: 개발 통합 브랜치
- **task/N**: 이슈 번호(N) 기반의 기능 개발 브랜치

### 절차
1. 이슈 생성
2. `task/N` 브랜치 생성
3. 구현 및 테스트 (Antigravity)
4. `develop` 브랜치로 Pull Request 및 머지
5. 최종 검증 후 `main` 반영

## 8. 기여 시 주의사항

1. **일관성 유지**: 기존 코드 스타일과 설계 패턴을 준수합니다.
2. **테스트 필수**: 모든 코드 변경은 테스트 코드를 동반해야 합니다.
3. **문서 동기화**: 코드 변경 시 관련 매뉴얼과 보고서도 즉시 업데이트합니다.
4. **한국어 사용**: 모든 문서는 특별한 요청이 없는 한 한국어로 작성합니다.
