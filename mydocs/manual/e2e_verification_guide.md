# 시스템 자동 검증 가이드 (E2E & Snapshot)

## 개요

본 프로젝트는 복잡한 로직이나 시각적 결과물을 검증하기 위해 **프로그래밍 방식의 자동 검증 체계**를 운영합니다. 이 문서는 시나리오 기반의 E2E 테스트와 결과물 기반의 스냅샷 테스트 운영 방법을 설명합니다.

```
시나리오 정의 → 자동 실행 → 상태/좌표 측정 → 규칙 기반 검증 → 결과 보고
```

## 1. E2E (End-to-End) 자동 검증

사용자의 입력이나 시스템의 상태 변화를 단계별로 시뮬레이션하고, 각 단계의 결과가 기대값과 일치하는지 확인합니다.

### 1.1 시나리오 작성 방식

JSON 또는 전용 스크립트 객체로 시스템 동작 순서를 선언합니다.

```javascript
const scenario = {
  name: 'TC-01: 기본 기능 흐름 검증',
  steps: [
    { type: 'input', value: '초기 데이터', label: 'init' },
    { type: 'action', command: 'process_data', label: 'process' },
    { type: 'verify', target: 'result_view', label: 'final' },
  ],
};
```

### 1.2 검증 규칙 (Expectations)

구조적 일관성뿐만 아니라, 세부적인 수치나 상태 규칙을 정의하여 검증합니다.

```javascript
const expectations = {
  // ── 상태 검증 ──
  recordCount: 1,
  status: 'COMPLETED',

  // ── 세부 규칙 검증 ──
  logic: [
    { rule: 'value-range', min: 0, max: 100 },
    { rule: 'integrity-check', compareWith: 'init' },
  ],
};
```

### 1.3 실행 결과 분석

- **성공**: 모든 단계와 규칙이 기대값 내에 있음
- **실패**: 불일치 발생 시 원인 분석 및 스크린샷/로그 확인
- **결과 보고**: CI 환경에서 자동으로 리포트(HTML/JSON) 생성

## 2. 스냅샷 회귀 검증 (Snapshot Testing)

"현재의 실행 결과가 이전(Golden Standard)과 바이트 단위로 동일한가"를 확인하여 의도치 않은 기능 저하를 방지합니다.

### 2.1 작동 원리

1. 샘플 입력을 기반으로 결과물을 생성합니다.
2. 저장소에 미리 저장된 **Golden 파일**과 현재 결과물을 비교합니다.
3. 차이가 발생하면 테스트가 실패하며, 개발자에게 확인을 요청합니다.

### 2.2 Golden 파일 갱신

결과물의 변경이 **의도된** 경우(버그 수정, 새 기능 도입 등) 다음 명령으로 Golden 파일을 갱신합니다.

```bash
# 예시 명령
UPDATE_GOLDEN=1 [test-command]
```

갱신 후에는 반드시 **결정성(Determinism) 확인**을 위해 테스트를 한 번 더 수행하여, 실행할 때마다 결과가 달라지지 않는지 확인해야 합니다.

## 3. 회귀 검증 체크리스트

코어 로직이나 엔진 수정을 포함한 PR을 병합할 때는 반드시 다음 절차를 거칩니다.

1. **로컬 전체 테스트 수행**: 스냅샷 테스트를 포함한 모든 테스트 통과 확인
2. **차이 분석**: 실패 시 변경된 부분이 의도한 대로인지(예: 레이아웃 미세 조정) 분석
3. **Golden 갱신 및 커밋**: 의도된 변경이라면 Golden 파일을 갱신하여 PR에 포함
4. **결정성 확인**: 같은 조건에서 항상 같은 결과가 나오는지 재확인

## 4. 향후 확장 계획

- **Visual Diff 도입**: 픽셀 단위 비교를 통한 시각적 오차 자동 감지
- **성능 벤치마크**: 실행 시간이나 메모리 사용량의 회귀 검증 추가
- **하드웨어 가속 검증**: 다양한 실행 환경(OS, 브라우저 엔진 등)에서의 일관성 확보
