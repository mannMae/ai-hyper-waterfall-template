# PR 리뷰 및 통합 워크플로우 매뉴얼

이 문서는 본 프로젝트에서 외부 기여자의 Pull Request(PR)를 처리하거나, 팀 내 협업 시 PR을 검토하고 통합하는 표준 절차를 정의합니다.

---

## 1. 개요

Hyper-Waterfall 방법론에 맞춰 **PR 처리도 표준화된 절차**로 운영합니다. 메인테이너 또는 검토 담당자는 본 매뉴얼의 순서에 따라 PR을 검증하고 통합해야 합니다.

## 2. PR 도착 시 확인 체크리스트

새로운 PR이 생성되면 다음 항목을 순서대로 확인합니다.

### 2.1 기본 메타 정보
- [ ] **Target 브랜치**: 개발용 통합 브랜치(`develop` 등)가 타깃인지 확인합니다.
- [ ] **연관 이슈**: PR 설명에 `Closes #N` 등 관련 이슈 참조가 포함되어야 합니다.
- [ ] **충돌 여부**: 자동 병합이 가능한 상태인지 확인합니다.
- [ ] **CI 상태**: 자동 빌드 및 테스트 통과 여부를 확인합니다.

### 2.2 규모 분석
- 변경된 라인 수와 파일 수를 파악하여 리뷰에 필요한 시간을 예측합니다.
- **대형 PR**: 기능별로 분할을 요청하거나 심층 검토 사이클을 운영합니다.
- **소형 PR**: 빠른 검토 후 즉시 통합을 고려합니다.

## 3. 리뷰 문서 작성

각 PR에 대해 검토 결과와 통합 계획을 문서화합니다.

### 3.1 리뷰 보고서 (`pr_{N}_review.md`)
- PR 요약 (작성자, 범위, 규모)
- 관련 이슈 및 요구사항 부합 여부
- 코드 품질 분석 (설계 준수, 중복 코드, 성능 등)
- 사전 검증 결과 (로컬 빌드 및 테스트 결과)
- 최종 권고 (승인, 수정 요청, 반려)

### 3.2 통합 계획서 (`pr_{N}_review_impl.md`)
- 병합 방식 (Merge, Squash & Merge, Rebase 등)
- 후속 작업 (문서 갱신, 추가 테스트, 마이그레이션 등)

## 4. 로컬 사전 검증

메인테이너는 PR 코드를 로컬 환경에서 직접 실행하여 검증합니다.

1. **브랜치 체크아웃**: PR 브랜치를 로컬로 가져옵니다.
2. **병합 시뮬레이션**: 타깃 브랜치와 병합 시 충돌이나 논리적 오류가 없는지 확인합니다.
3. **빌드 및 테스트**: 로컬 환경에서 전체 빌드와 테스트를 수행합니다.
4. **회귀 테스트**: 기존 기능에 영향이 없는지 확인합니다.

## 5. 승인 및 피드백

### 5.1 수정 요청 시 (Feedback Loop)
- 수정이 필요한 이유를 구체적으로 설명합니다.
- 대안이나 참고 코드를 제공하여 기여자를 돕습니다.
- 톤앤매너: 전문적이고 존중하는 태도를 유지하되, 품질 기준은 엄격히 적용합니다.

### 5.2 승인 및 통합
- 모든 조건이 충족되면 최종 승인 코멘트를 남기고 통합을 진행합니다.

## 6. 후속 처리 (필수 절차)

### 6.1 이슈 종료 확인
- PR 병합 후 연관된 이슈가 정상적으로 닫혔는지 확인합니다. 필요시 수동으로 종료합니다.

### 6.2 기여자 감사 인사
- 기여에 대한 감사를 표하고, 변경 사항이 프로젝트에 미친 긍정적 영향(성능 개선, 버그 해결 등)을 언급합니다.

### 6.3 로컬 환경 동기화
- 통합된 코드를 로컬 브랜치에 반영하고, 임시 검증 브랜치를 정리합니다.

### 6.4 결과 기록
- 오늘 할 일(`mydocs/orders/`) 문서에 PR 처리 내역을 기록합니다.

## 7. 예외 케이스 처리

- **보안 이슈**: 보안 취약점이 발견된 경우 즉시 비공개로 전환하거나 수정을 요청합니다.
- **대규모 아키텍처 변경**: 단독 판단이 어려운 경우 프로젝트 오너나 커뮤니티와 상의합니다.
- **의존성 업데이트**: 자동화된 봇(Dependabot 등)의 PR은 정기적으로 검토하여 통합합니다.

## 8. 결론

PR 리뷰는 단순히 코드를 검사하는 과정을 넘어, 프로젝트의 품질을 유지하고 협업 문화를 구축하는 핵심 단계입니다. 모든 과정은 **문서화**되어야 하며, 투명하게 관리되어야 합니다.
