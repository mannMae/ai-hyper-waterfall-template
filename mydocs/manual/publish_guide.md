# 배포 및 릴리즈 가이드

본 프로젝트의 배포 대상, 절차 및 버전 관리 정책을 안내합니다. 모든 배포 과정은 자동화를 지향하며, Hyper-Waterfall의 품질 관문을 통과해야 합니다.

## 1. 배포 대상 (Deployment Targets)

| 대상 | 플랫폼 | 배포 방식 | 트리거 |
|------|---------|----------|--------|
| 웹 애플리케이션 | [GitHub Pages / Vercel] | CI/CD 자동 | `main` 브랜치 업데이트 |
| 패키지/라이브러리 | [NPM / PyPI / Cargo] | CI/CD 자동 | GitHub Release 생성 (태그) |
| 애플리케이션/익스텐션 | [App Store / Marketplace] | CI/CD 자동 | GitHub Release 생성 (태그) |

## 2. 버전 관리 정책 (Versioning)

본 프로젝트는 **Semantic Versioning (SemVer)**을 준수합니다.

- **MAJOR (v1.x.x)**: 하위 호환성이 깨지는 대규모 변경
- **MINOR (vx.1.x)**: 하위 호환성을 유지하는 새로운 기능 추가
- **PATCH (vx.x.1)**: 하위 호환성을 유지하는 버그 수정 및 최적화

### 버전 동기화
- 다중 패키지 프로젝트의 경우, 코어 라이브러리의 버전을 기준으로 다른 패키지들의 버전을 동기화합니다.
- `Cargo.toml`, `package.json` 등 메타데이터 파일의 버전을 일관되게 관리합니다.

## 3. 배포 워크플로우 (CI/CD)

### 3.1 자동 워크플로우
- **CI (Build & Test)**: 모든 Push와 PR 발생 시 자동으로 빌드 및 테스트를 수행하여 무결성을 검증합니다.
- **CD (Deploy)**: 검증된 코드가 특정 브랜치에 병합되거나 릴리즈 태그가 생성되면 지정된 플랫폼으로 자동 배포됩니다.

### 3.2 배포 흐름도
```
코드 구현 완료 (Task Branch)
  ↓
PR 생성 → CI 자동 검증 (Build + Test + Lint)
  ↓
승인 후 Merge (develop/main)
  ↓
GitHub Release 생성 (vX.X.X)
  ↓ 배포 워크플로우 자동 실행
  ├─ 아티팩트 빌드 (WASM, Binary, Assets)
  ├─ 패키지 저장소 배포 (NPM, etc.)
  └─ 서비스 환경 업데이트 (Web, Cloud)
```

## 4. 배포 절차 (Release Steps)

### 1단계: 최종 검증
- 모든 단위 테스트 및 통합 테스트 통과 확인
- E2E 시나리오 및 시각적 검증(Snapshot) 완료

### 2단계: 메타데이터 업데이트
- 버전 번호 업데이트 (`package.json`, `Cargo.toml` 등)
- 변경 이력(`CHANGELOG.md`) 작성 및 최신화

### 3단계: 커밋 및 푸시
- 변경 사항을 커밋하고 메인 브랜치에 반영합니다.

### 4단계: 릴리즈 생성
- GitHub 또는 협업 도구에서 새 Release를 생성하고 태그(예: `v1.0.0`)를 부여합니다.
- 릴리즈 노트에 주요 변경 사항과 수정된 버그 목록을 포함합니다.

### 5단계: 배포 확인
- 배포 워크플로우의 실행 로그를 모니터링하여 성공 여부를 확인합니다.
- 실제 서비스 환경에서 정상 동작하는지 최종 점검합니다.

## 5. 시크릿 및 토큰 관리 (Secrets)

배포에 필요한 인증 정보는 안전하게 관리되어야 합니다.

- **NPM_TOKEN**: 패키지 배포용
- **AWS/GCP Keys**: 클라우드 인프라 접근용
- **APP_STORE_API_KEY**: 앱 스토어 배포용

---

> [!CAUTION]
> 수동 배포는 가급적 지양하며, 자동화된 파이프라인 실패 시에만 예외적으로 수행합니다. 수동 배포 시에도 버전 충돌 및 권한 이슈를 사전에 점검해야 합니다.
